# EVO-0006 - Consolidate Cloudflare Project Foundation

## Epic

`Cloudflare Infrastructure`

## Objective

Consolidate the active Cloudflare project foundation for CityQuest so the monorepo supports `Cloudflare Workers`, future `Cloudflare Pages`, future `Cloudflare D1`, and later `Cloudflare R2` through a consistent and maintainable setup.

This task is about tooling, environment conventions, naming, documentation, and cleanup of the active path. It must not implement business logic, authentication, database schema, production deployment automation, or real product endpoints.

## Why This Task Exists

CityQuest no longer treats AWS as the active MVP direction. The repository already contains a Worker-based API foundation and Cloudflare-first ADRs, but the project still needs one explicit task that:

- consolidates the Cloudflare path as the active infrastructure direction,
- defines naming and environment conventions clearly,
- documents how local Cloudflare development is expected to work,
- prepares later `Workers`, `Pages`, `D1`, and `R2` tasks without redesign,
- and removes confusion created by legacy AWS-oriented assumptions in active documents and workflows.

This EVO is therefore not a greenfield platform bootstrap. It is a consolidation and cleanup task aligned with the current state of the repository.

## Functional Context

CityQuest is a cultural tourism platform with a gamified exploration loop. The MVP focuses on the route `Jaen: Echoes of Stone`, where users explore real heritage locations, receive clues, identify visual targets, take photos, validate progress, and unlock narrative content.

The Cloudflare platform supports the technical foundation of that MVP:

- `Cloudflare Workers` for the backend API,
- `Cloudflare D1` for destinations, routes, POIs, progress, users, and analytics,
- `Cloudflare Pages` for the admin panel and possibly a public landing,
- `Cloudflare R2` later for controlled multimedia assets such as images and audio.

This task prepares and clarifies that foundation only.

## Technical Goal

Create or refine a Cloudflare configuration layer for the monorepo that:

- uses `Wrangler` as the main Cloudflare development and deployment tool,
- supports local development cleanly,
- defines clear environment conventions,
- avoids hardcoded secrets,
- prepares the project for `Workers`, `Pages`, and `D1`,
- documents how the Cloudflare project is structured,
- keeps Cloudflare-specific configuration explicit and easy to maintain,
- and avoids presenting AWS tooling as part of the active implementation path.

## Scope

This task should include:

1. validate and refine `Wrangler` configuration for the Cloudflare backend workspace;
2. define Cloudflare environment naming conventions;
3. define resource naming conventions for `Workers`, `Pages`, `D1`, and future `R2`;
4. add or adapt monorepo scripts for Cloudflare-oriented commands where they improve clarity;
5. prepare local development environment examples;
6. document how to authenticate with Cloudflare locally;
7. document how the Cloudflare project is expected to be structured;
8. ensure Cloudflare configuration does not include secrets or account-specific sensitive values;
9. prepare the repository for future `D1` and `Pages` tasks without provisioning them here;
10. remove or explicitly de-activate obsolete AWS assumptions from the active setup path.

## Recommended Project Conventions

The Cloudflare project should use predictable names so future resources are easy to identify.

Recommended names:

```txt
cityquest-dev-api
cityquest-dev-admin
cityquest-dev-db
cityquest-dev-assets
```

Future environments can follow the same pattern:

```txt
cityquest-staging-api
cityquest-staging-admin
cityquest-staging-db
cityquest-prod-api
cityquest-prod-admin
cityquest-prod-db
```

For the MVP, only the `dev` environment needs to be actively prepared.

## Expected Cloudflare Tooling

The active project path should use:

```txt
Wrangler
Cloudflare Workers
Cloudflare Pages
Cloudflare D1
R2 later only
```

Do not introduce new active dependencies on:

```txt
AWS CDK
Serverless Framework for AWS
SST
API Gateway
Lambda
DynamoDB
Cognito
S3
CloudFront
CloudWatch
```

Historical AWS groundwork may remain in the repository as legacy context, but it must not be presented as the current implementation path for the MVP.

## Expected Configuration Areas

Depending on the monorepo structure, configuration will likely live in areas such as:

```txt
apps/
  api/
    wrangler.jsonc or wrangler.toml
    .dev.vars.example
apps/
  admin/
    ...
docs/
  cloudflare-setup.md
```

If a root-level Cloudflare helper is useful, it may be added, but the Worker configuration should remain directly understandable from `apps/api`.

## Environment Configuration Guidance

The task should prepare example environment files only.

Example:

```txt
apps/api/.dev.vars.example
```

Possible non-secret example values:

```txt
CITYQUEST_API_NAME=cityquest-api
CITYQUEST_APP_ENV=development
CITYQUEST_LOG_LEVEL=info
```

Do not commit real secrets.

Do not commit:

```txt
CLOUDFLARE_API_TOKEN
CLOUDFLARE_ACCOUNT_ID
OPENAI_API_KEY
JWT_SECRET
SESSION_SECRET
```

If these are needed later, document where they should be configured, but do not add real values.

## Expected Scripts

The root monorepo may expose scripts similar to:

```json
{
  "scripts": {
    "cf:login": "wrangler login",
    "cf:whoami": "wrangler whoami",
    "api:dev": "npm run dev --workspace @cityquest/api",
    "api:deploy": "npm run deploy --workspace @cityquest/api"
  }
}
```

The exact workspace syntax can vary. The important point is that a developer can understand how to run and validate Cloudflare tooling from the repository.

## Expected Documentation

Add or refine a short Cloudflare setup document, for example:

```txt
docs/cloudflare-setup.md
```

It should explain:

- why CityQuest uses Cloudflare instead of AWS for the MVP,
- which Cloudflare services are part of the architecture,
- how to log in with Wrangler,
- how to run the Worker locally,
- what naming convention is used,
- which resources are planned for later tasks,
- how secrets should be handled,
- which AWS artifacts remain as historical context only,
- and what is intentionally out of scope for this EVO.

The document should stay practical and short.

## Expected Result

At the end of this task, the repository should be clearly ready for Cloudflare-based development.

A developer should be able to understand and perform commands such as:

```bash
npm install
npm run cf:whoami
npm run api:dev
```

or equivalent commands supported by the final repo scripts.

The project should clearly communicate that Cloudflare is the active infrastructure direction for the MVP and that AWS is legacy context only.

## Acceptance Criteria

- `Wrangler` is available and configured for the active backend project.
- The active cloud setup is Cloudflare-oriented.
- The repository contains clear Cloudflare naming conventions.
- The repository contains clear environment conventions, at least for `dev`.
- The active setup path does not require AWS infrastructure steps for MVP progress.
- No new AWS CDK setup is required for the current MVP direction.
- No new AWS Lambda, API Gateway, DynamoDB, Cognito, S3, CloudFront, or CloudWatch configuration is introduced.
- Cloudflare configuration does not include real secrets.
- Example environment files exist where useful.
- Local Cloudflare development is documented.
- The future role of `Workers`, `Pages`, `D1`, and `R2` is documented.
- `R2` is documented as future storage only, not required for the current setup.
- The task does not provision `D1` schema yet.
- The task does not implement product endpoints.
- The task does not implement authentication.
- The task does not implement CI/CD.
- The setup remains compatible with the existing monorepo structure.
- Active documentation and scripts no longer imply that AWS is the next path to continue.

## Out of Scope

Do not include in this task:

- implementing Worker API routes,
- implementing `GET /health`,
- creating real `D1` schema tables,
- writing SQL migrations,
- inserting seed data,
- implementing authentication,
- creating user or admin roles,
- creating the demo account,
- deploying Cloudflare Pages,
- configuring production domains,
- creating `R2` buckets,
- uploading multimedia assets,
- setting up GitHub Actions deployment,
- implementing analytics,
- implementing validation logic,
- integrating AI providers.

## Definition of Done

This EVO is complete when the CityQuest monorepo has one clear Cloudflare project foundation, `Wrangler` is configured coherently, environment and naming conventions are documented, local Cloudflare development is understandable, and future tasks can safely build `Workers`, `Pages`, and `D1` functionality on top of this setup without reworking the infrastructure direction.
