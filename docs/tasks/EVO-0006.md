# EVO-0006 - Add API Gateway Foundation with AWS CDK

## Objective

Add the first public API entry point for the CityQuest serverless backend through AWS CDK.

This task is about infrastructure foundation, not product endpoint logic. The result should be a deployable API Gateway base that future Lambda-backed routes can attach to without reworking the stack design.

## Why This Task Exists

API Gateway will become the public ingress layer for:

- the mobile app
- the admin panel
- future internal tools and testing flows

This task introduces the first visible serverless architecture component in AWS and creates the public base URL that later backend tasks will build on.

## Decision Summary

The task should lock the following defaults:

- use `HTTP API`, not `REST API`
- target `dev` only
- enable basic development CORS
- expose the base API URL through a CDK output
- no Cognito or other authentication yet
- no authorizers yet
- no WAF
- no custom domain
- no throttling policy work yet
- no business routes yet

## Implementation Approach

The implementation should create or extend the CDK application under `infra/` and define a dedicated stack for the API foundation.

Expected approach:

- keep the stack simple and Lambda-ready
- define the API entirely through infrastructure as code
- use clear, environment-aware naming
- prepare the stack so Lambda routes can be attached in later tasks without redesign

Naming defaults:

- API name example: `cityquest-dev-api`
- stack name should also be explicit and environment-aware, for example a recognizable `dev` API or platform stack name

This task should not attach business endpoints yet. It only needs to establish the public API surface and the stack structure that future tasks will extend.

## CDK Design

The CDK implementation should:

- use `aws-cdk-lib/aws-apigatewayv2`
- create one `HTTP API`
- create a `dev` stage suitable for early testing
- configure permissive development CORS for known local development origins
- export one output named like `CityQuestApiUrl`

Expected CORS baseline:

- allow local admin origins
- allow browser-based testing tools
- allow the HTTP methods likely needed in the near term
- allow common JSON and future auth-related headers

Important note:

- mobile development traffic usually does not depend on browser CORS
- the configuration should still support the local admin panel and browser testing cleanly from the beginning

## Expected Repo Changes

The implementation will likely touch:

- `infra/` CDK application and stack files
- package manifests or scripts if CDK tooling is not initialized yet
- documentation if the infrastructure structure needs to be explained
- an ADR if the HTTP API decision is not already captured elsewhere

The structure should remain intentionally simple. This task should prepare the API for future Lambda integration, not overbuild the entire cloud platform in one step.

## AWS Prerequisites

Before deployment, the following external prerequisites should be in place:

- AWS credentials configured locally
- the correct AWS region selected
- CDK bootstrap completed for the target account and region if it has not already been done

The deployment flow should not require manual AWS Console setup outside the standard CDK bootstrap and credential setup.

## Manual Validation

After implementation, validate the task with these checks:

- `cdk synth` succeeds
- `cdk deploy` succeeds
- the API appears in AWS
- the deployment output includes the base API URL
- the output format looks like `https://xxxxx.execute-api.<region>.amazonaws.com/`

It is acceptable if the base URL does not return useful product content yet. Functional responses can arrive in later tasks once routes such as a healthcheck are attached.

## Acceptance Criteria

- API Gateway is defined from CDK
- the API is created through infrastructure as code
- `HTTP API` is used unless a documented reason changes that
- the API name is recognizable, for example `cityquest-dev-api`
- the base URL is exported as a stack output
- CORS is enabled for development use cases
- the structure is ready to connect Lambda routes
- no authentication is implemented yet
- no business routes are created yet
- `cdk synth` works after the change
- `cdk deploy` can deploy the API without complex manual intervention

## Out of Scope

This task must not include:

- `/destinations`, `/routes`, or other business endpoints
- Cognito or other authentication work
- authorizers
- WAF
- advanced throttling
- custom domains
- a production stage
- advanced observability setup
- Lambda integrations beyond structural readiness

## Follow-up Dependencies

This task is meant to unblock:

- Lambda route integration
- healthcheck endpoint wiring
- later protected admin routes with Cognito or another authorizer strategy

## Important Interfaces and Outputs

The task specification assumes these important outputs:

- a recognizable logical API identity for `dev`
- a CDK output named like `CityQuestApiUrl`
- deployment against an AWS account and region already configured outside the repo
- a stack design that allows future Lambda route attachment without changing API type

## Test and Validation Scenarios

The implementation should be considered ready when these scenarios are covered:

- `cdk deploy` succeeds once AWS authentication and CDK bootstrap are ready
- the base URL is visible in terminal output and CDK outputs
- the API exists in the AWS Console

- future protected admin routes remain possible without replacing the API type

## AWS Execution Handoff

This section is intended to help a human operator complete the AWS side of the task later, especially with the help of another assistant such as ChatGPT.

### What the Current Code Already Does

The repository already contains the CDK code for this task.

Current implementation summary:

- the CDK app entry point is `infra/bin/cityquest-infra.ts`
- the environment configuration is created in `infra/src/config/createInfraEnvironmentConfig.ts`
- the stack is defined in `infra/src/stacks/CityQuestApiFoundationStack.ts`
- the reusable API construct is defined in `infra/src/constructs/CityQuestHttpApiFoundation.ts`
- the development CORS policy is defined in `infra/src/factories/createDevelopmentCorsPreflightOptions.ts`

At synth and deploy time, the current implementation creates:

- one API Gateway `HTTP API`
- one `dev` stage with `autoDeploy: true`
- one CloudFormation output named `CityQuestApiUrl`

### Current Naming and Environment Rules

The current code assumes:

- environment name: `dev`
- API name: `cityquest-dev-api`
- stack name: `CityQuestDevApiFoundationStack`
- default region fallback: `eu-west-1`

Region resolution order in the code is:

1. CDK context value `region`
2. environment variable `CITYQUEST_AWS_REGION`
3. environment variable `CDK_DEFAULT_REGION`
4. fallback `eu-west-1`

### Current CORS Configuration

The current API is configured for development-focused CORS only.

Allowed origins:

- `http://localhost:3000`
- `http://127.0.0.1:3000`
- `http://localhost:3001`
- `http://127.0.0.1:3001`
- `http://localhost:5173`
- `http://127.0.0.1:5173`

Allowed methods:

- `GET`
- `POST`
- `PUT`
- `PATCH`
- `DELETE`
- `OPTIONS`

Allowed headers:

- `authorization`
- `content-type`
- `x-requested-with`

### What Must Be Ready in AWS

Before running `cdk deploy`, make sure:

- you are authenticated against the correct AWS account
- the correct AWS region is selected
- CDK bootstrap has been completed for that account and region

Typical prerequisites:

- AWS CLI installed
- AWS credentials configured with a profile or environment variables
- enough permissions to create API Gateway, CloudFormation, IAM-related bootstrap resources, and CDK deployment assets

### Recommended AWS Region for This Project

If there is no stronger reason to choose another region yet, use:

- `eu-west-1`

That matches the current code fallback and is a sensible default for this project.

### Suggested Deployment Flow

Recommended step-by-step flow:

1. confirm which AWS account you want to use
2. confirm the region, ideally `eu-west-1`
3. bootstrap CDK if this account and region have not been bootstrapped yet
4. run `cdk synth`
5. run `cdk diff`
6. run `cdk deploy`
7. capture the output value `CityQuestApiUrl`
8. verify the API exists in API Gateway

Example command style:

```bash
npm run bootstrap --workspace @cityquest/infra -- --region eu-west-1
npm run synth --workspace @cityquest/infra -- --context region=eu-west-1
npm run diff --workspace @cityquest/infra -- --context region=eu-west-1
npm run deploy --workspace @cityquest/infra -- --context region=eu-west-1
```

If the AWS profile matters, the operator can provide it through the AWS CLI environment or profile selection before running those commands.

### What Output to Expect

After a successful deployment, CloudFormation/CDK should emit:

- `CityQuestApiUrl`

Expected shape:

```text
https://xxxxx.execute-api.eu-west-1.amazonaws.com/
```

The exact hostname will be generated by AWS.

### What Is Normal After Deployment

The API may exist and still not return useful product data yet.

That is expected because this task does not create:

- business routes
- Lambda integrations
- healthcheck endpoint wiring
- authentication

So success for this task means:

- the API exists
- the stage exists
- the output exists
- the infrastructure is ready for the next backend-routing task
