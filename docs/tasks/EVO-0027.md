# EVO-0027 - Create the Mobile Route Detail Screen

## Epic

`Mobile MVP`

## Objective

Implement the first real route detail screen in `apps/mobile` by replacing the temporary local
route fixture with live Worker API data from:

```txt
GET /routes/jaen-echoes-of-stone
```

This EVO should turn the current route detail placeholder into a real pre-gameplay review screen
that the user can inspect before entering the current objective flow.

## Why This Task Exists

The repository already has:

- a working Expo mobile workspace
- a minimal internal mobile navigation stack from `EVO-0024`
- a live destination selector from `EVO-0026`
- a public Worker endpoint for route detail from `EVO-0021`
- seeded route content for:
  - `Jaén: Echoes of Stone`
  - `Cathedral of Jaén`
  - `Arab Baths of Jaén`
  - first visual objectives for both POIs

That means the next useful mobile slice is no longer just listing the playable destination.
The app now needs to show the real route content that exists behind the selected destination.

## Functional Context

CityQuest is a gamified cultural tourism platform.

At this stage of the MVP, after choosing `Jaén`, the user should be able to review the route they
are about to play:

- route title
- route description
- difficulty
- estimated duration
- seeded POIs

The backlog also mentions `modo demo/real`, but the repository does not yet have a complete demo
or validation flow behind that label.

Practical interpretation:

- this EVO may expose a lightweight UI treatment for the current route mode
- but it must not pretend that gameplay validation, progress, or auth-backed mode switching already
  exists if it does not

## Current Repository Reality

At the time of this EVO:

- `apps/mobile` still uses a local adapter for `RouteDetailScreen`
- `apps/api` already exposes `GET /routes/:routeSlug`
- the route detail endpoint already returns:
  - route metadata
  - destination metadata
  - POIs
  - objectives
- hints and unlockable content are not yet part of the mobile route detail flow

That means this task should focus on:

- replacing the local mobile route detail fixture with a real API read
- defining a mobile-facing route detail view model
- keeping the current navigation flow intact

## Technical Goal

Implement a mobile route detail screen that:

- fetches live route detail from the Worker API
- renders the real seeded route data for `jaen-echoes-of-stone`
- shows route metadata and POI summaries in a clear mobile layout
- lets the user continue into the current objective flow

This should remain a lightweight Expo-native screen, not a full gameplay controller.

## Scope

This task should include:

1. add a mobile API read path for `GET /routes/:routeSlug`;
2. replace the temporary `RouteDetailScreen` fixture with live route data;
3. define a view model for route, destination, POIs, and first objective linkage;
4. show route title, description, difficulty, estimated duration, and POI list;
5. preserve the ability to continue into the current objective screen;
6. decide a minimal and honest treatment for `demo/real` in the UI if shown at all;
7. ensure mobile typecheck and lint still pass.

## Non-Goals

This EVO must not:

- implement objective validation
- implement progress tracking
- implement camera flow
- implement hints
- implement unlockable content delivery
- implement authentication
- redesign the whole mobile navigation stack
- add route editing or admin behaviors

## UX Guidance

The route detail screen should feel like a pre-play screen, not a raw JSON projection.

Reasonable UI elements:

- route title
- short route description
- destination name
- difficulty
- estimated duration
- list of POIs
- CTA to continue into the current objective flow

For `demo/real`:

- if shown, it should be framed as a lightweight local UI indicator only
- do not imply that full demo validation behavior already exists
- do not block this EVO on a complete mode system if the backend does not yet support one

## Data Guidance

Recommended real data source:

```txt
GET /routes/:routeSlug
```

Expected first supported slug:

```txt
jaen-echoes-of-stone
```

Recommended mobile behavior:

- fetch route detail using the slug passed through navigation
- render the live API data directly through a small mobile-specific mapping layer
- keep any route-mode badge or label local until a fuller mode system exists

## Integration Guidance

This EVO should build directly on:

- `EVO-0021` for backend route detail
- `EVO-0024` for navigation
- `EVO-0026` for destination selection

Reasonable implementation targets inside `apps/mobile`:

```txt
src/
  screens/
    RouteDetailScreen.tsx
  services/
    currentMvpCatalog.ts         # likely reduced further
    routes/
      getRouteDetail.ts          # or equivalent mobile API adapter
  navigation/
    AppNavigator.tsx
```

Do not overengineer this:

- no heavy query library unless clearly needed
- no global state library
- no speculative route repository abstraction

## Expected Result

At the end of this task:

- the route detail screen is no longer backed only by a local fixture
- `Jaén: Echoes of Stone` loads from the Worker API
- the user can review real route information before entering gameplay
- the current objective flow still remains reachable

## Acceptance Criteria

- the mobile app reads route detail from `GET /routes/:routeSlug`
- `jaen-echoes-of-stone` loads successfully in the mobile route detail screen
- the screen shows real route metadata from the API
- the screen shows seeded POIs from the API
- the implementation does not fake validation, progress, or unlock logic
- any `demo/real` treatment remains honest and lightweight
- mobile lint and typecheck pass
- the implementation stays aligned with the current Expo foundation

## Out of Scope

Do not include in this task:

- gameplay validation
- objective completion
- hints
- unlocked content
- persistence of route progress
- authentication
- backend changes to expand the route detail contract beyond what already exists

## Definition of Done

This EVO is complete when the CityQuest mobile app can load and render the real route detail for
`Jaén: Echoes of Stone` from the Worker API, preserve the current navigation into gameplay, and do
so without pretending that later gameplay systems already exist.

## Suggested Implementation Order

1. review the current `RouteDetailScreen` and the local route fixture
2. add the minimal mobile API adapter for `GET /routes/:routeSlug`
3. define the route detail view model
4. update the screen UI for real route metadata and POIs
5. keep navigation into the current objective screen working
6. validate with mobile typecheck, lint, and manual UI verification

## Backlog Note

This EVO should treat the route detail screen as the bridge between selection and gameplay.

If a later backlog item wants:

- real demo-vs-real gameplay mode
- objective validation
- hints
- unlockable content

those should build on the route detail contract and UI introduced here rather than being implied
prematurely by the screen copy in this EVO.
