# EVO-0007 - Consolidate Worker HTTP API Foundation

## Epic

`Cloudflare Infrastructure`

## Objective

Consolidate the HTTP API foundation of the active CityQuest backend in `apps/api`.

This task is not about creating the first Worker runtime from scratch. That foundation already exists. This EVO should refine the current Worker transport layer so it has a clearer routing model, centralized HTTP response handling, centralized error handling, and centralized CORS behavior that future endpoints can reuse safely.

The result should be a cleaner and more extensible HTTP base for later endpoint work such as `GET /health`, destinations, routes, progress, validation, admin flows, and D1-backed handlers.

## Why This Task Exists

CityQuest already has a working Cloudflare Worker foundation:

- a Worker-native `fetch` entrypoint
- a minimal initialization response at `/`
- controlled `404` behavior
- local Wrangler execution
- unit and integration tests

That is enough for the runtime foundation, but not yet enough for a durable API transport layer.

Before adding real backend endpoints, the repository needs:

- a clearer API/router composition layer
- reusable JSON response conventions
- reusable JSON error conventions
- centralized CORS handling
- predictable handling for unsupported routes and methods

Without this step, later endpoint work would likely duplicate transport code or force avoidable refactors across multiple EVOs.

## Functional Context

CityQuest is a gamified cultural tourism platform whose MVP is centered on:

- route exploration
- visual objectives
- validation through image and location
- narrative content unlocks
- progression

The Worker backend will later support concerns such as:

- health and readiness checks
- destinations and route retrieval
- route progress
- objective validation
- analytics event ingestion
- demo mode support
- future admin operations

This EVO does not implement those features. It only prepares the transport-level HTTP base they will sit on.

## Current Repository Reality

At the time of this EVO:

- `apps/api` is already the active backend workspace
- the Worker runtime is already working locally
- the root path `/` already returns an initialization-style JSON response
- request handling is already separated to some extent, but remains minimal
- no real API route family exists yet

Practical interpretation:

- do not treat this task as “create the Worker”
- treat it as “harden and structure the Worker HTTP layer”

## Technical Goal

Refine the existing `apps/api` transport layer so it:

- keeps `src/index.ts` extremely thin
- routes requests through a dedicated HTTP composition layer
- defines reusable JSON success and error helpers
- centralizes CORS policy and preflight handling
- supports explicit unknown-route behavior
- supports explicit unsupported-method behavior where relevant
- remains framework-light and easy to reason about
- stays compatible with future D1, auth, and business endpoint work

## Scope

This task should include:

1. refine the API routing structure for the Worker;
2. create or improve a central API/router composition layer;
3. define route grouping conventions for future endpoint families;
4. define standard JSON success response helpers;
5. define standard JSON error response helpers;
6. centralize local-development CORS behavior;
7. add centralized `OPTIONS` preflight handling;
8. add explicit unknown-route handling;
9. add explicit unsupported-method handling where useful;
10. keep the Worker entrypoint clean by delegating transport concerns;
11. add or refine HTTP-oriented types/utilities only where they clearly help;
12. preserve the existing initialization route behavior unless there is a strong reason to change it;
13. add or update tests to cover the HTTP foundation changes.

## Non-Goals

This EVO must not:

- implement `GET /health`
- add destinations, routes, progress, validation, analytics, or admin endpoints
- add authentication
- add D1 bindings or D1 access logic
- add R2 integration
- introduce CI/CD concerns
- introduce production CORS hardening
- introduce a traditional Node server

## Recommended Implementation Style

Use the native Worker runtime and a small manual router.

This is the preferred approach because:

- the current API surface is still tiny
- the repo already has a working Worker foundation
- adding a routing framework now would increase architectural surface area without a proven need

### Preferred Approach

- keep `fetch(request, env, ctx)` as the runtime entry
- delegate request processing into a small composition layer
- match routes manually
- keep transport helpers explicit and easy to test

### Not Recommended for This EVO

- introducing Express
- introducing a long-running Node server
- introducing Hono unless there is a clear, documented reason to do so

If a framework is ever introduced later, that should be a deliberate technical decision, not an incidental convenience inside this EVO.

## Recommended Structure

The exact folder layout should follow the current repo patterns rather than forcing a cosmetic rewrite.

An acceptable evolution from the existing structure would be something like:

```txt
apps/
```

Notes:

- do not restructure the whole workspace just to match a theoretical template
- prefer incremental improvement over directory churn
- keep routing and HTTP concerns clearly separate from application services

## Routing Guidance

This EVO should establish conventions for future routes, not implement them.

Future public route families will likely include:

```txt
GET    /health
GET    /destinations
GET    /routes/:routeId
GET    /routes/:routeId/progress
POST   /routes/:routeId/start
POST   /objectives/:objectiveId/validate
POST   /objectives/:objectiveId/hint
GET    /content/:contentId
POST   /analytics/events
```

Future admin route families will likely include:

```txt
POST   /admin/destinations
PUT    /admin/destinations/:destinationId
POST   /admin/routes
PUT    /admin/routes/:routeId
POST   /admin/pois
PUT    /admin/pois/:poiId
POST   /admin/objectives
PUT    /admin/objectives/:objectiveId
POST   /admin/content
PUT    /admin/content/:contentId
GET    /admin/analytics
```

This EVO should only prepare conventions so these route groups can be added later without redesigning the transport layer.

## Response Guidance

The API should move toward predictable JSON conventions.

Recommended success shape for future endpoint-style responses:

```json
{
  "success": true,
  "data": {},
  "meta": {}
}
```

The current initialization response at `/` may remain intentionally lightweight for now if that keeps the existing runtime foundation stable.

Recommended error shape:

```json
{
  "success": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "Route not found"
  }
}
```

Rules:

- do not expose stack traces
- do not expose internal implementation details
- keep error payloads stable and testable

## CORS Guidance

The API should include centralized development-oriented CORS support.

Minimum expected behavior:

- handle `OPTIONS` preflight requests
- include `Access-Control-Allow-Origin`
- include allowed methods
- include allowed headers
- avoid duplicating CORS logic across route handlers

This EVO should not attempt to finalize production CORS policy.

## Expected Result

At the end of this task:

- the Worker still runs locally through Wrangler
- `/` still behaves intentionally
- unknown routes return controlled JSON errors
- preflight requests are handled centrally
- route handling is easier to extend
- future endpoint EVOs can focus on business behavior instead of transport cleanup

Example unknown route behavior:

```bash
curl http://localhost:8787/unknown
```

Possible response:

```json
{
  "success": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "Route not found"
  }
}
```

## Acceptance Criteria

- `apps/api` remains the active backend workspace
- the Worker entrypoint delegates transport handling through a clearer API/router layer
- a central HTTP routing/composition path exists
- standard JSON response helpers exist, or equivalent response conventions are clearly centralized
- standard JSON error helpers exist for transport-level failures such as unknown routes
- unknown routes return controlled JSON errors
- unsupported methods are handled intentionally where relevant
- centralized CORS handling exists
- `OPTIONS` preflight requests are handled
- the Worker still runs locally through Wrangler
- the implementation remains Worker-native
- the implementation does not depend on AWS Lambda, API Gateway, Express, or a long-running Node server
- the implementation does not add D1 access yet
- the implementation does not add product endpoints yet
- the implementation does not add authentication yet
- tests cover the refined HTTP foundation sufficiently
- future route-extension expectations are documented briefly in code or README-level docs

## Out of Scope

Do not include in this task:

- formal `GET /health`
- D1 binding configuration
- D1 schema or migrations
- seed data
- authentication
- user/admin roles
- demo account
- destinations endpoint
- route detail endpoint
- route progress endpoint
- validation endpoints
- analytics endpoints
- admin CRUD endpoints
- R2 integration

- production CORS hardening
- custom domains

## Definition of Done

This EVO is complete when the Cloudflare Worker backend has a cleaner and more extensible HTTP transport foundation, with centralized routing, response/error conventions, and CORS/preflight handling, while preserving the current Worker runtime and avoiding business logic or persistence concerns.

## Suggested Implementation Order

1. review the current `apps/api` transport structure
2. refine the API composition/router layer
3. keep `src/index.ts` thin
4. introduce centralized success/error response helpers
5. introduce centralized CORS and preflight handling
6. add intentional unknown-route and method behavior
7. update integration tests for HTTP transport behavior
8. run local Worker validation and existing test/build scripts

## Backlog Note

This EVO should be treated as the HTTP transport refinement step for the existing Worker foundation.

If another backlog item still describes “creating the first minimal Worker backend,” that item should be reconsidered or re-scoped, because the runtime foundation already exists before this EVO starts.
