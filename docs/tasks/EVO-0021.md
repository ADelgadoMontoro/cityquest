# EVO-0021 - Implement the Public Route Detail Endpoint in the Worker API

## Epic

`Backend MVP`

## Objective

Implement the first public route detail endpoint in `apps/api` so the Worker can return the real seeded route data for `Jaén: Echoes of Stone`, including its POIs and first published visual objectives.

This EVO should build directly on the content already stored in `Cloudflare D1` and expose it through a stable read contract that mobile and admin clients can consume.

The endpoint must only expose data that already exists in the repository and database baseline.

## Why This Task Exists

The repository already has:

- a stable Worker HTTP foundation
- a public `GET /destinations` endpoint
- a real D1 binding
- an initial content schema
- a seeded route baseline for:
  - `Jaén: Echoes of Stone`
  - `Cathedral of Jaén`
  - `Arab Baths of Jaén`
  - first visual objectives for both POIs

That means the next useful backend slice is no longer just listing destinations. The mobile app and future admin read views now need the full route structure for the seeded MVP experience.

Before route detail screens or gameplay screens can consume real backend content, the project needs one public route detail endpoint.

## Functional Context

CityQuest is a gamified cultural tourism platform.

The current MVP route is:

- `Jaén: Echoes of Stone`

The current real route detail content already present in D1 includes:

- route metadata
- POIs:
  - `Cathedral of Jaén`
  - `Arab Baths of Jaén`
- first visual objectives for each POI

This EVO should expose that route detail slice as read-only API data.

It must not fabricate:

- progress state
- validation state
- hints not yet seeded
- unlockable content not yet seeded
- future POIs not yet stored

## Current Repository Reality

At the time of this EVO:

- `apps/api` is the active backend workspace
- the Worker can already list destinations from D1
- D1 already contains a real route baseline and seeded POIs/objectives
- hints and unlockable content still exist in schema but have not yet been seeded

That means this task should focus on:

- joining or composing route, POI, and objective reads
- shaping a stable nested JSON contract
- preserving the current Worker-native and D1-native approach

## Technical Goal

Implement a public endpoint for route detail retrieval.

Recommended route:

```txt
GET /routes/:routeSlug
```

Preferred first real path for the current seed:

```txt
GET /routes/jaen-echoes-of-stone
```

The endpoint should:

- read the published route from D1
- include its published POIs ordered by `display_order`
- include each POI’s published visual objectives ordered by `display_order`
- return a stable JSON success shape

## Scope

This task should include:

1. add a public route detail route to the Worker router;
2. add the minimal D1 read path needed for the seeded route, POIs, and objectives;
3. define a stable nested response shape;
4. include only real seeded and published data;
5. add or update integration tests for the endpoint;
6. keep the implementation Worker-native and D1-native;
7. ensure build and test flows still pass.

## Non-Goals

This EVO must not:

- implement progress retrieval
- implement route start logic
- implement objective validation
- expose hints if they are not yet seeded
- expose unlockable content if it is not yet seeded
- add write endpoints
- add authentication
- add admin-only projections
- redesign the schema

## Endpoint Guidance

Recommended route:

```txt
GET /routes/:routeSlug
```

For the current MVP seed, the first supported route should be:

```txt
GET /routes/jaen-echoes-of-stone
```

Recommended behavior:

- return the route only if it is `published`
- return only `published` POIs
- return only `published` visual objectives
- preserve deterministic ordering
- return `404` if the route is unknown or unpublished

## Response Guidance

Recommended minimum payload direction:

```json
{
  "success": true,
  "data": {
    "route": {
      "id": "route-jaen-echoes-of-stone",
      "slug": "jaen-echoes-of-stone",
      "title": "Jaén: Echoes of Stone",
      "description": "...",
      "status": "published",
      "difficulty": "easy",
      "estimatedDurationMinutes": 300,
      "displayOrder": 0,
      "destination": {
        "id": "destination-jaen",
        "slug": "jaen",
        "name": "Jaén"
      },
      "pois": [
        {
          "id": "poi-catedral-de-jaen",
          "slug": "catedral-de-jaen",
          "name": "Cathedral of Jaén",
          "indoorMode": 1,
          "displayOrder": 0,
          "objectives": [
            {
              "id": "objective-catedral-de-jaen-estatua-san-fernando",
              "slug": "estatua-san-fernando",
              "title": "Statue of Saint Ferdinand",
              "targetType": "statue",
              "indoorMode": 0,
              "displayOrder": 0
            }
          ]
        }
      ]
    }
  },
  "meta": {}
}
```

This is direction, not a strict literal response, but the response should remain aligned with the JSON transport conventions already used by the Worker.

## Data Contract Guidance

Prefer an explicit read model over leaking raw SQL rows into the transport layer.

Recommended shape:

- one route read model
- one POI read model
- one objective read model

Keep it pragmatic:

- no ORM
- no generic projection framework
- no speculative DTO hierarchy beyond what this endpoint really needs

## Query Guidance

Keep the read path explicit and small.

Preferred approach:

- one focused service or a small composition of read helpers
- route lookup first
- then POI read
- then objective read

It is acceptable to start with multiple clear queries if that keeps the code more obvious than one oversized joined query.

Prioritize clarity and testability over premature optimization.

## Expected Result

At the end of this task:

- `GET /routes/jaen-echoes-of-stone` exists
- it reads from D1 successfully
- it returns the seeded route, POIs, and objectives already present in the repository
- mobile/admin work can start consuming a real route detail endpoint

## Acceptance Criteria

- the Worker exposes a public route detail endpoint
- the endpoint reads the seeded route from D1
- the endpoint includes seeded POIs in deterministic order
- the endpoint includes seeded visual objectives in deterministic order
- the endpoint returns only real published data
- the endpoint returns `404` for unknown or unsupported route slugs
- the endpoint is covered by integration tests
- the implementation remains Worker-native and D1-native
- existing build and test flows still pass

## Out of Scope

Do not include in this task:

- route progress
- route start
- objective validation
- hints if they are not yet seeded
- unlockable content if it is not yet seeded
- analytics
- admin editing flows
- authentication

## Definition of Done

This EVO is complete when the Worker can return the real seeded route detail for `Jaén: Echoes of Stone`, including its current POIs and objectives, through a stable public endpoint with test coverage and without inventing missing product data.
