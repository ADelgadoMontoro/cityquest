## Seed Strategy Guidance

Preferred options:

### Option A: Seed as a numbered SQL migration

Use a new SQL file after the existing schema migrations.

Pros:

- keeps everything explicit and versioned
- works well with the current migration-first D1 strategy
- easy to replay in local and remote environments

Cons:

- seed data becomes part of the migration history and should be treated as versioned product baseline

### Option B: Separate seed script

Use a separate seed command/script outside the migration sequence.

Pros:

- more flexible for future reseeding workflows

Cons:

- adds an extra operational path earlier than necessary

### Recommended MVP Choice

Prefer a numbered SQL migration unless a strong operational reason appears to separate seeds from migrations this early.

This keeps the repository simpler and aligned with the current D1 strategy.
