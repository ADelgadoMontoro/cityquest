# EVO-0020 - Implement the Public Destinations Listing Endpoint in the Worker API

## Epic

`Backend MVP`

## Objective

Implement the first real public read endpoint in `apps/api` to list available destinations from `Cloudflare D1`.

This EVO should turn the seeded D1 content into an actual Worker response that later mobile and admin clients can consume.

The endpoint must be based on the real seeded data already present in the repository, not on invented placeholders.

## Why This Task Exists

The repository already has:

- a working Cloudflare Worker backend
- centralized HTTP routing and JSON conventions
- a real D1 binding
- an initial content schema
- real seeded content for:
  - destination `Jaén`
  - route `Jaén: Echoes of Stone`
  - POIs `Cathedral of Jaén` and `Arab Baths of Jaén`

That means the next high-value step is no longer more platform setup. It is exposing real content through a public API contract.

Before mobile route browsing or admin read views can consume real backend data, the project needs a first public listing endpoint.

## Functional Context

CityQuest is a gamified cultural tourism platform.

At this point in the MVP, the real destination baseline is:

- `Jaén`

Future destinations such as `Úbeda`, `Baeza`, or `Cazorla` may exist as product ideas or roadmap items, but they are not currently seeded in D1 and must not be fabricated in this EVO.

Practical interpretation:

- the API should list what is really in the database
- if the product later wants “locked future destinations,” those should come from real stored records or an explicit product decision in a later EVO

## Current Repository Reality

At the time of this EVO:

- `apps/api` is the active backend workspace
- the Worker HTTP layer is already stable
- `GET /health` already exists
- D1 already contains a real destination baseline
- no public business read endpoints exist yet

That means this task should focus on:

- querying D1
- mapping rows into a stable JSON contract
- integrating the route into the existing Worker router
- testing the endpoint end-to-end

## Technical Goal

Implement a public endpoint for destination listing.

Recommended route:

```txt
GET /destinations
```

The endpoint should:

- read published destinations from D1
- return a stable JSON success shape
- remain compatible with the Worker-native routing foundation
- avoid exposing internal SQL or persistence details

## Scope

This task should include:

1. add a destination listing route to the Worker router;
2. add the minimal query/service path needed to read destinations from D1;
3. define the response shape for the listing;
4. return only real stored data from the current D1 baseline;
5. add or update integration tests for the endpoint;
6. keep the implementation Worker-native and D1-native;
7. ensure build and test flows still pass.

## Non-Goals

This EVO must not:

- implement route detail retrieval
- implement POI detail retrieval
- fabricate blocked future destinations not present in D1
- add write endpoints
- add admin endpoints
- add authentication
- add progress or validation logic
- add R2 integration
- redesign the migration strategy

## Endpoint Guidance

Recommended route:

```txt
GET /destinations
```

Recommended behavior:

- return destinations ordered by `display_order`
- include only `published` destinations by default
- include fields that are already real and meaningful in the schema

Recommended minimum payload direction:

```json
{
  "success": true,
  "data": {
    "destinations": [
      {
        "id": "destination-jaen",
        "slug": "jaen",
        "name": "Jaén",
        "description": "...",
        "status": "published",
        "coverImageUrl": null,
        "displayOrder": 0
      }
    ]
  },
  "meta": {
    "count": 1
  }
}
```

This is direction, not a mandatory final literal payload, but the response should stay consistent with the existing HTTP conventions introduced earlier.

## Data Contract Guidance

The implementation should prefer a small explicit read model over leaking raw SQLite row naming directly into the transport layer.

That means:

- database columns may remain snake_case internally
- API response fields may be mapped into the project’s transport naming convention if the existing API layer prefers that

Use the simplest approach that stays clear and testable.

## Query Guidance

Keep the first D1 read path simple.

Preferred approach:

- a focused query function or repository-like helper inside `apps/api`
- a small service or handler adapter that turns the result into the public response

Do not introduce:

- a full ORM
- speculative repository hierarchies
- generic query abstraction layers

## Expected Result

At the end of this task:

- `GET /destinations` exists
- it reads from D1 successfully
- it returns the real destination baseline already seeded in the repository
- mobile/admin work can start consuming a real public content endpoint

## Acceptance Criteria

- the Worker exposes a public `GET /destinations` endpoint
- the endpoint reads from D1
- the endpoint returns the seeded `Jaén` destination
- the endpoint returns a stable JSON response
- the endpoint is covered by integration tests
- the endpoint does not fabricate future blocked destinations
- the implementation remains Worker-native and D1-native
- existing build and test flows still pass

## Out of Scope

Do not include in this task:

- route detail
- POI detail
- objective detail
- destination creation or editing
- admin-only filtering
- authentication
- teaser or placeholder destinations not stored in D1

## Definition of Done

This EVO is complete when the Worker can return the real published destinations currently stored in D1 through a stable public `GET /destinations` endpoint, with test coverage and without inventing future content.

## Suggested Implementation Order

1. review the current Worker router and D1 typing
2. define the public response shape
3. add the destination query path
4. add the route handler
5. add integration coverage
6. validate with build, test, and local Worker execution

## Backlog Note

This EVO should expose only what the repository already knows to be real content.

If the product later wants a mixed response such as:

- published destinations
- future blocked destinations

then those blocked items should come from real seeded records or a later explicit product decision, not from hardcoded placeholders hidden inside the endpoint.
