# EVO-0026 - Create the Mobile Province Selector Screen

## Epic

`Mobile MVP`

## Objective

Implement the first real destination selection screen in `apps/mobile`, replacing the temporary
local destination listing with live Worker API data for published destinations while also showing
the next planned provinces as locked MVP options.

This EVO should make the mobile selector feel like a real product screen instead of a navigation
placeholder.

## Why This Task Exists

The repository already has:

- a working Expo mobile workspace
- a minimal internal navigation stack from `EVO-0024`
- a public `GET /destinations` endpoint in `apps/api`
- a real seeded destination baseline in D1:
  - `Jaén`
- a clear MVP roadmap that still mentions future destinations such as:
  - `Úbeda`
  - `Baeza`
  - `Cazorla`

That means the next useful mobile slice is no longer just “can the app move between screens?”
It is “can the app render a real destination selector grounded in the actual backend state?”

## Functional Context

CityQuest is a gamified cultural tourism platform.

At this stage of the MVP:

- `Jaén` is the only real published destination already stored in D1
- `Úbeda`, `Baeza`, and `Cazorla` are still roadmap destinations, not live backend content

So the selector has to communicate two different kinds of availability:

- real, playable destination:
  - `Jaén`
- visible but locked future destinations:
  - `Úbeda`
  - `Baeza`
  - `Cazorla`

Practical interpretation:

- `Jaén` should come from the real backend
- locked future destinations should remain an explicit mobile UI decision in this EVO
- the app must not pretend those locked items already exist in D1

## Current Repository Reality

At the time of this EVO:

- `apps/mobile` already has a welcome screen and navigation flow
- the current `DestinationsScreen` still reads from a temporary in-memory adapter
- the API already exposes `GET /destinations`
- the API currently returns only real stored data

That means this task should focus on:

- replacing the mobile destination fixture with a real API read
- defining a mobile-friendly selector view model
- mixing real and locked items in the screen without polluting backend contracts

## Technical Goal

Implement a mobile destination selector screen that:

- fetches published destinations from the Worker API
- renders the real available destination list
- appends a small locked roadmap section for `Úbeda`, `Baeza`, and `Cazorla`
- lets the user continue only through the real playable destination

This EVO should keep the current navigation foundation but make the destination step data-driven.

## Scope

This task should include:

1. add a mobile API read path for `GET /destinations`;
2. replace the temporary destination fixture currently used by the selector;
3. define a selector view model that can represent:
   - available destinations
   - locked future destinations;
4. render `Jaén` as selectable from live backend data;
5. render `Úbeda`, `Baeza`, and `Cazorla` as locked entries in the mobile UI;
6. keep the implementation Expo-native and lightweight;
7. ensure mobile typecheck and lint still pass.

## Non-Goals

This EVO must not:

- change the backend `GET /destinations` contract to fabricate locked items
- seed `Úbeda`, `Baeza`, or `Cazorla` into D1
- add route detail fetching
- add authentication
- add progress logic
- add camera or validation
- redesign navigation around a third-party routing framework

## UX Guidance

The selector should clearly distinguish:

- playable now
- coming soon / locked

Reasonable behaviors:

- `Jaén` appears as the primary selectable card
- locked items appear visually disabled or clearly marked as unavailable
- selecting a locked destination should do nothing or show a very lightweight “coming soon” state

Keep it simple and intentional.

## Data Guidance

Recommended real data source:

```txt
GET /destinations
```

Recommended UI composition:

- fetch real destinations from API
- map them into mobile selector items
- append static locked roadmap items locally inside `apps/mobile`

Example direction:

```ts
[
  {
    slug: 'jaen',
    name: 'Jaén',
    availability: 'available',
  },
  {
    slug: 'ubeda',
    name: 'Úbeda',
    availability: 'locked',
  },
]
```

The important rule is:

- only `available` items should come from the real API
- `locked` items should be clearly tagged as mobile-side roadmap placeholders

## Integration Guidance

This EVO should build directly on `EVO-0024`.

Reasonable implementation targets inside `apps/mobile`:

```txt
src/
  screens/
    DestinationsScreen.tsx
  services/
    currentMvpCatalog.ts        # likely replaced or reduced
    destinations/
      listDestinations.ts       # or equivalent mobile API adapter
  navigation/
    AppNavigator.tsx
```

Do not overengineer this:

- no global state library
- no heavy data-fetching framework unless already needed
- no repository architecture for the mobile app yet

## Expected Result

At the end of this task:

- the destination selector is no longer backed only by a local fixture
- `Jaén` is loaded from the real Worker endpoint
- `Úbeda`, `Baeza`, and `Cazorla` appear as locked future destinations
- the user can continue from the selector into the current playable route path

## Acceptance Criteria

- the mobile app reads real destination data from `GET /destinations`
- the selector shows `Jaén` as available
- the selector shows `Úbeda`, `Baeza`, and `Cazorla` as locked
- the locked items are clearly not treated as real live destinations
- selecting `Jaén` still leads into the current MVP route flow
- mobile lint and typecheck pass
- the implementation stays lightweight and aligned with the current Expo foundation

## Out of Scope

Do not include in this task:

- route detail consumption
- route detail redesign
- current objective gameplay
- hints
- unlockable content
- backend changes to support locked destinations
- new D1 seeds for future provinces

## Definition of Done

This EVO is complete when the CityQuest mobile app shows a real destination selector that loads
the current playable destination from the Worker API, keeps future provinces visibly locked in the
UI, and remains consistent with the current navigation base introduced in `EVO-0024`.

## Suggested Implementation Order

1. review the current `DestinationsScreen` and the temporary mobile fixture
2. add the minimal mobile API adapter for `GET /destinations`
3. define a selector view model for available vs locked entries
4. update the screen UI and selection behavior
5. validate navigation still works for `Jaén`
6. run mobile typecheck, lint, and manual UI verification

## Backlog Note

This EVO should keep a clear boundary between:

- real backend content
- roadmap UI affordances

If a later backlog item wants `Úbeda`, `Baeza`, or `Cazorla` to become real playable destinations,
that should happen through:

- new seeds
- new route content
- real API reads

not by silently upgrading the mobile placeholders introduced here.
