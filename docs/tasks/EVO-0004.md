# EVO-0004 - Initialize Cloudflare Workers Backend Foundation

## Epic

`Setup and Core Architecture`

## Objective

Migrate the current backend foundation in `apps/api` from its AWS Lambda-oriented direction to a `Cloudflare Workers` foundation using TypeScript.

This task is about backend runtime foundation, not business endpoint implementation. The result should be a clean Worker-ready backend base that can later support healthchecks, mobile endpoints, admin endpoints, D1 access, authentication, analytics, and validation flows without redesigning the architecture.

## Why This Task Exists

CityQuest has adopted a Cloudflare-first MVP platform direction to reduce infrastructure cost risk and simplify early deployment.

The backend still needs a public HTTP runtime for:

- the mobile app
- the admin panel
- future internal tools
- local and browser-based manual testing

This task establishes the first active Cloudflare-native backend runtime and replaces the existing Lambda-oriented path as the preferred implementation direction for the MVP.

## Functional Context

CityQuest is a cultural tourism platform with a gamified exploration loop. The MVP focuses on the route `Jaen: Echoes of Stone`, where users explore real heritage locations, receive clues, identify visual targets, take photos, validate progress, and unlock narrative content.

The backend will later be responsible for:

- exposing REST endpoints for the mobile app and admin panel
- reading and writing structured data in Cloudflare D1
- managing user progression
- validating objectives
- delivering unlocked content
- recording analytics events
- supporting a demo account
- protecting future admin routes

This EVO prepares only the technical base.

## Technical Goal

Create a minimal backend foundation in `apps/api` that:

- uses `Cloudflare Workers` with TypeScript
- runs locally with `Wrangler`
- is deployable later through the Cloudflare Worker workflow
- keeps a professional and maintainable folder structure
- is ready for future HTTP route expansion
- can accept future D1, R2, and KV bindings without forcing them now
- does not depend on AWS runtime services for the active backend path
- does not depend on Express or a traditional long-running Node server

## Scope

This task should include:

1. Rework the backend runtime in `apps/api` to target Cloudflare Workers.
2. Add and configure Wrangler for local development and future deploys.
3. Adapt TypeScript configuration for the Worker runtime.
4. Create a Worker entry point based on the standard `fetch` interface.
5. Prepare a clean structure for future routes, services, repositories, middlewares, types, and utilities.
6. Replace Lambda-specific runtime adapters with Worker-oriented transport code.
7. Add or adapt development and validation scripts.
8. Add minimal local run documentation.
9. Verify that the Worker can start locally.

## Recommended Structure

The backend already exists, so this task should evolve it rather than recreate it from scratch.

A reasonable target structure is:

```txt
apps/
  api/
    src/
      index.ts
      bootstrap/
      routes/
      services/
      repositories/
      middlewares/
      types/
      utils/
    tests/
      unit/
      integration/
    wrangler.toml
    package.json
    tsconfig.json
    README.md
```

### Folder Notes

- `src/index.ts`: Worker entry point.
- `bootstrap/`: composition and runtime assembly.
- `routes/`: future HTTP route definitions or route handlers.
- `services/`: application services.
- `repositories/`: future D1 access layer.
- `middlewares/`: future auth, validation, or request-level concerns.
- `types/`: backend-specific contracts and runtime types.
- `utils/`: reusable helpers.

Not every folder needs real business logic yet, but the layout should be ready for later tasks.

## Runtime and Design Guidance

The Worker implementation should:

- expose one runtime entry point using `fetch(request, env, ctx)`
- keep transport concerns separate from application logic
- define a typed `Env` contract for future bindings
- keep error handling explicit
- remain simple enough for TDD-driven iteration

Design patterns should be applied pragmatically where they add value. For this task, the main useful patterns are:

- `composition root` for runtime assembly
- `factory` where environment-aware configuration improves clarity
- `adapter` between Worker transport and application services

This task should **not** force a routing framework decision unless the implementation clearly needs it. If a framework such as `Hono` is later introduced, it should be treated as a specific technical decision and reflected in an ADR when appropriate.

## Expected Scripts

The `apps/api/package.json` workspace should expose scripts equivalent to:

```json
{
  "scripts": {
    "dev": "wrangler dev",
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "deploy": "wrangler deploy"
  }
}
```

The exact script implementation may wrap `Wrangler` through local helper scripts if that improves cross-environment stability, but the backend workspace itself must remain directly operable through the Worker workflow.

## Expected Result

At the end of this task, `apps/api` should contain a minimal Cloudflare Worker backend that can be started locally.

The expected local flow should look like:

```bash
cd apps/api
npm install
npm run dev
```

And a basic request such as:

```bash
curl http://localhost:8787/
```

should return a minimal response confirming that the backend runtime is initialized.

Example response:

```json
{
  "environment": "development",
  "service": "cityquest-api",
  "status": "initialized"
}
```

This response is only a runtime verification placeholder. It is not yet a real product endpoint.

## Acceptance Criteria

- `apps/api` exists as the active backend workspace.
- the active backend runtime is configured for `Cloudflare Workers`
- the Worker is implemented in TypeScript
- Wrangler configuration exists
- a Worker entry point exists in `src/index.ts`
- the Worker can run locally with `wrangler dev`
- the active backend path no longer depends on AWS Lambda, API Gateway, or a traditional Node server
- the backend does not use Express
- the workspace includes an initial structure for routes, services, repositories, middlewares, types, and utilities
- a development script exists for local Worker execution
- a typecheck script exists
- tests remain part of the backend validation workflow
- the initial response contains no real business logic
- there is no real D1 integration yet
- there is no authentication yet
- there are no real MVP business endpoints yet
- no secrets or hardcoded credentials are committed
- minimal documentation explains how to run the backend locally

## Out of Scope

Do not include in this task:

- a formal `GET /health` endpoint
- D1 schema creation or SQL migrations
- real D1 access logic
- R2 integration
- KV-backed product logic
- authentication
- user and admin roles
- demo account implementation
- destinations endpoints
- routes endpoints
- user progression logic
- GPS validation
- image validation
- analytics implementation
- CI/CD
- production deployment strategy
- custom domains
- AI integration

## Definition of Done

This EVO is complete when the backend foundation in `apps/api` has been successfully reoriented to Cloudflare Workers, compiles correctly, runs locally through Wrangler, remains testable, and fits coherently into the monorepo as the new preferred backend runtime path for the MVP.
