# EVO-0015 - Implement Worker Healthcheck Endpoint

## Epic

`Backend MVP`

## Objective

Implement a formal `GET /health` endpoint in the active CityQuest Cloudflare Worker backend.

This task should validate that the Worker HTTP API foundation is usable for real endpoints after the transport layer consolidation. The endpoint must provide a simple, stable, JSON-based health response that can be used for local testing, deployment checks, CI smoke tests, and future monitoring.

This EVO is not about implementing business functionality. It should only add a backend healthcheck endpoint using the existing Worker-native API structure.

## Why This Task Exists

CityQuest needs a reliable way to verify that the backend API is alive and responding correctly before adding product endpoints such as destinations, routes, progress, validation, analytics, or admin operations.

The project already has a Cloudflare Worker foundation and should now expose a formal healthcheck endpoint that confirms:

* the Worker runtime is responding,
* the HTTP routing layer works,
* JSON response conventions are usable,
* errors and unsupported methods are handled predictably,
* the API can be checked manually or automatically.

This is the first intentionally defined backend endpoint.

## Functional Context

CityQuest is a gamified cultural tourism platform. The MVP focuses on `Jaén: Ecos de Piedra`, where users explore heritage locations, identify visual objectives, validate progress through image and location, and unlock narrative content.

The backend will later support:

* destination listing,
* route details,
* route progress,
* objective validation,
* content unlocks,
* analytics events,
* demo mode,
* admin operations.

This EVO does not implement those features. It only adds the healthcheck endpoint that confirms the API foundation is ready for them.

## Current Repository Reality

At the time of this EVO:

* `apps/api` is already the active backend workspace.
* The backend runs as a Cloudflare Worker.
* The Worker already has a native `fetch` entrypoint.
* The root path `/` may already return a minimal initialization response.
* EVO-0007 should have consolidated routing, responses, errors, CORS and preflight handling.
* Unit and integration tests should already exist or be part of the backend validation workflow.

Practical interpretation:

* do not create a Worker from scratch,
* do not redesign the routing layer,
* do not add a routing framework,
* use the existing HTTP composition/router layer.

## Technical Goal

Add a formal `GET /health` endpoint that:

* runs inside the existing Cloudflare Worker backend,
* uses the existing routing/composition layer,
* returns a stable JSON response,
* follows the API response conventions established in EVO-0007 where appropriate,
* does not depend on D1, R2, authentication, external APIs, or product data,
* can be tested locally through Wrangler,
* is covered by integration tests.

## Endpoint

```txt
GET /health
```

## Expected Response

Recommended response:

```json
{
  "success": true,
  "data": {
    "status": "ok",
    "service": "cityquest-api",
    "environment": "development"
  },
  "meta": {
    "timestamp": "2026-06-03T12:00:00.000Z"
  }
}
```

The exact `environment` value can come from a non-sensitive environment variable, for example:

```txt
APP_ENV=development
```

If the current response convention after EVO-0007 is intentionally simpler, this is also acceptable:

```json
{
  "status": "ok",
  "service": "cityquest-api",
  "environment": "development",
  "timestamp": "2026-06-03T12:00:00.000Z"
}
```

The important point is that the response is stable, JSON-based, safe, and testable.

## Scope

This task should include:

1. Add route registration for `GET /health`.
2. Implement a dedicated health route handler.
3. Use the existing JSON response helper if available.
4. Return a stable health payload.
5. Include the service name.
6. Include the current environment if available.
7. Include a dynamic timestamp if useful.
8. Ensure the endpoint returns HTTP `200`.
9. Ensure unsupported methods on `/health` return intentional behavior.
10. Ensure CORS and `OPTIONS` behavior still work through the centralized HTTP layer.
11. Add or update tests for the health endpoint.
12. Update minimal backend documentation if useful.

## Implementation Guidance

The healthcheck handler should be intentionally simple.

It should not:

* query a database,
* validate authentication,
* inspect user state,
* call external services,
* depend on image validation,
* depend on analytics,
* depend on admin logic.

The healthcheck is a runtime/API availability check, not a deep readiness check.

A future `/ready` endpoint could check D1 bindings or migrations if needed, but that is out of scope for this EVO.

## Method Handling Guidance

`GET /health` should return `200`.

If another method is used, for example:

```txt
POST /health
```

the API should return intentional unsupported-method behavior.

Recommended response:

```json
{
  "success": false,
  "error": {
    "code": "METHOD_NOT_ALLOWED",
    "message": "Method not allowed"
  }
}
```

Recommended status:

```txt
405 Method Not Allowed
```

If the current router handles unsupported methods globally, use that behavior instead of duplicating method logic inside the health handler.

## CORS Guidance

CORS should be handled by the centralized layer created or refined in EVO-0007.

Expected behavior:

* `OPTIONS /health` should return a valid preflight response.
* `GET /health` should include the expected CORS headers if the current API convention applies them globally.
* The health handler should not manually duplicate CORS headers unless the current architecture requires it.

## Tests

Add or update tests to cover the endpoint.

Recommended integration test cases:

* `GET /health` returns `200`.
* `GET /health` returns valid JSON.
* response includes `status: "ok"` or equivalent.
* response includes `service: "cityquest-api"`.
* unknown route behavior still works after adding `/health`.
* unsupported method behavior is intentional.
* `OPTIONS /health` is handled if preflight tests already exist.

Recommended unit tests, if useful:

* health handler returns the expected payload shape.
* environment fallback works if no environment variable is provided.

Do not over-test implementation details. Prefer testing observable API behavior.

## Expected Local Validation

The endpoint should work through Wrangler.

Example:

```bash
cd apps/api
npm run dev
```

Then:

```bash
curl -i http://localhost:8787/health
```

Expected result:

```txt
HTTP/1.1 200 OK
Content-Type: application/json
```

With a body similar to:

```json
{
  "success": true,
  "data": {
    "status": "ok",
    "service": "cityquest-api",
    "environment": "development"
  }
}
```

The exact timestamp value should not be hardcoded.

## Acceptance Criteria

* `GET /health` exists in the Cloudflare Worker API.
* The endpoint is registered through the existing API/router layer.
* The Worker entrypoint remains thin.
* The endpoint returns HTTP `200`.
* The endpoint returns valid JSON.
* The response includes a clear service identifier, preferably `cityquest-api`.
* The response includes a health status, preferably `ok`.
* The response does not expose secrets, tokens, internal stack traces, Cloudflare account data, or sensitive environment values.
* The endpoint does not require authentication.
* The endpoint does not access D1.
* The endpoint does not call external services.
* Unsupported methods on `/health` are handled intentionally.
* `OPTIONS /health` is handled consistently with the centralized CORS/preflight behavior.
* Existing `/` behavior is preserved unless there is a deliberate reason to change it.
* Existing unknown-route behavior still works.
* Tests cover the health endpoint behavior.
* Local execution through Wrangler still works.
* Typecheck and test scripts pass.

## Definition of Done

This EVO is complete when the active Cloudflare Worker backend exposes a formal `GET /health` endpoint that responds with a safe, stable JSON health payload, is registered through the consolidated HTTP API foundation, is covered by tests, and can be validated locally with Wrangler without introducing business logic, persistence, authentication, or external service dependencies.

## Suggested Implementation Order

1. Review the routing/composition layer created or refined in EVO-0007.
2. Add a dedicated health route handler.
3. Register `GET /health` in the API router.
4. Use existing response helpers for the success payload.
5. Ensure unsupported methods are handled intentionally.
6. Validate `OPTIONS /health` through centralized CORS behavior.
7. Add or update integration tests.
8. Run typecheck and tests.
9. Start the Worker locally with Wrangler.
10. Validate with `curl http://localhost:8787/health`.
