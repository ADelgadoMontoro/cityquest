# EVO-0009 - Provision and Bind the Development D1 Database

## Epic

`Cloudflare Infrastructure`

## Objective

Provision the first real `Cloudflare D1` database for the CityQuest development environment and bind it to the active Worker backend in `apps/api`.

This EVO is not about schema design, repositories, or business persistence yet. It is about making the development database exist for real, wiring it into the Worker configuration, extending the Worker runtime contract, and preparing the repository for later migration work.

The result should be a real D1 baseline that later tasks such as `EVO-0070` can build on without needing to revisit platform wiring.

## Why This Task Exists

CityQuest already has:

- a Cloudflare-first MVP direction
- a functioning Worker runtime
- a consolidated Worker HTTP transport layer
- a formal `GET /health` endpoint

That means the current bottleneck is no longer transport or runtime setup. The next missing platform capability is structured persistence.

Before implementing endpoints for destinations, routes, content, progress, validation, or analytics, the project needs:

- a real D1 database in the `dev` environment
- a stable Worker binding for that database
- a typed `env.DB` contract
- an obvious place for future SQL migrations
- minimal operational documentation for the D1 workflow

Without this step, later backend work would either be blocked or forced to mix infrastructure setup with feature delivery.

## Functional Context

CityQuest is a gamified cultural tourism platform whose MVP will eventually need persistent data for:

- destinations
- routes
- POIs
- objectives
- unlockable content
- progress
- achievements
- analytics inputs
- future admin-managed content

This EVO does not create those tables or records. It only provisions the development database and prepares the Worker/runtime contract that future tasks will use.

## Current Repository Reality

At the time of this EVO:

- `apps/api` is already the active backend workspace
- the Worker runtime already exists and runs locally
- routing, JSON transport behavior, and `GET /health` already exist
- the repository already uses `Wrangler` through a local wrapper flow
- `apps/api/wrangler.toml` already contains the active Worker config
- `src/types/worker.ts` already defines the Worker environment contract
- no D1 binding exists yet
- no D1 schema should exist yet

Practical interpretation:

- do not treat this task as “introduce Cloudflare” from scratch
- do not add application tables here
- do not add repositories here
- do not add data-access logic here
- do not fake D1 with placeholder-only configuration
- provision the actual dev D1 resource and bind it properly

## Technical Goal

Provision the real development D1 database and bind it cleanly into the active Worker backend so future code can access it through a stable Worker binding:

```ts
env.DB
```

This EVO should:

- make the D1 resource real
- make the Worker aware of it
- make the TypeScript environment contract aware of it
- leave a clear location for future migrations
- document the operating flow used by this repository
- keep the current Worker foundation stable after the binding is introduced

## Scope

This task should include:

1. create the real Cloudflare D1 database for the `dev` environment;
2. choose and confirm a stable development database name;
3. add the D1 binding to the active Worker configuration in `apps/api/wrangler.toml`;
4. update the Worker environment type so `env.DB` is part of the contract;
5. prepare a clear folder for future SQL migrations inside `apps/api`;
6. add minimal documentation for D1 create/list/migration workflows;
7. ensure the Worker still runs locally after the binding is introduced;
8. ensure existing build and test flows still pass;
9. avoid committing secrets, tokens, or unrelated account-sensitive material.

## Naming Guidance

Recommended development database name:

```txt
cityquest-dev-db
```

Recommended Worker binding name:

```txt
DB
```

This keeps future Worker code simple and conventional:

```ts
env.DB
```

Future environments should follow the same pattern:

```txt
cityquest-staging-db
cityquest-prod-db
```

For now, only the development database needs to be provisioned.

## Repository Integration Guidance

The important repository-level outcome is not only “a D1 exists in Cloudflare,” but also that the repo is structurally ready for later schema work.

Reasonable integration targets:

```txt
apps/
  api/
    wrangler.toml
    migrations/
      README.md
    src/
      types/
        worker.ts
```

Rules:

- keep D1 ownership close to the active backend workspace
- prefer `apps/api/migrations/` over introducing a second infrastructure path
- do not add speculative repository or ORM abstractions yet

## Wrangler and Binding Guidance

The Worker should declare a real D1 binding in `apps/api/wrangler.toml`.

Expected shape:

```toml
[[d1_databases]]
binding = "DB"
database_name = "cityquest-dev-db"
database_id = "<real-cloudflare-database-id>"
```

Important rules:

- use the real Cloudflare-generated database ID
- do not invent placeholder IDs
- committing the real dev `database_id` is acceptable if it represents the actual development environment for this repository
- do not commit API tokens, account tokens, or other secrets

## Worker Type Guidance

The Worker environment contract should be updated to include the D1 binding.

Example direction:

```ts
type WorkerRuntimeEnv = {
  DB: D1Database;
  CITYQUEST_API_NAME?: string;
  CITYQUEST_APP_ENV?: string;
  CITYQUEST_LOG_LEVEL?: string;
};
```

Preferred end state:

- `DB` is part of the stable Worker contract
- later feature work does not need to renegotiate the binding name

Do not add fake repository interfaces or placeholder data services just to “use” the new type in this EVO.

## Wrapper and Local Workflow Guidance

This repository already wraps `Wrangler` through local scripts in `apps/api/scripts/`.

That matters because D1 provisioning and verification should stay compatible with the repo’s actual workflow instead of assuming global `wrangler` state.

Typical commands should align with the current wrapper flow, for example:

```bash
npm run cf:whoami
npm run login --workspace @cityquest/api
npm run whoami --workspace @cityquest/api
```

For D1 operations, the documentation and implementation should stay consistent with the current workspace-oriented Worker setup. Do not document a separate ad-hoc path that ignores the repository wrapper model.

## Operational Guidance

Typical real commands will likely include:

```bash
npm exec wrangler d1 create cityquest-dev-db --workspace @cityquest/api
npm exec wrangler d1 list --workspace @cityquest/api
```

Future migration work should be able to build on commands conceptually like:

```bash
npm exec wrangler d1 migrations apply cityquest-dev-db --local --workspace @cityquest/api
npm exec wrangler d1 migrations apply cityquest-dev-db --remote --workspace @cityquest/api
```

Actual schema migrations belong to `EVO-0070`, not this EVO.
