# EVO-0070 - Define the Initial MVP Content Schema in D1

## Epic

`Modelo de Datos y Gestión de Contenido`

## Objective

Define and implement the first real SQL schema for the CityQuest MVP in `Cloudflare D1`.

This EVO should create the initial content-oriented database structure for the project, based on the MVP route and exploration model already described across the repo. It should define the first migration, the core relational tables, the basic constraints, and the indexes needed to support future content retrieval endpoints.

This task is about schema definition only. It must not insert product seed data, implement repositories, create endpoints, or add business logic.

## Why This Task Exists

CityQuest already has:

- a Cloudflare-first active platform direction
- a functioning Worker runtime
- a consolidated Worker HTTP layer
- a formal `GET /health` endpoint
- a real D1 database provisioned and bound to the Worker as `env.DB`
- a migrations location under `apps/api`

That means the next useful step is no longer infrastructure wiring. The next useful step is defining the first real application data structure.

Before implementing endpoints that expose destinations, routes, POIs, objectives, hints, or unlockable content, the project needs:

- a first migration file
- a clear core content schema
- stable table names and foreign-key paths
- basic indexes for the main read patterns

Without this step, later endpoint work would either be blocked or forced to invent the schema inside feature tasks.

## Functional Context

CityQuest is a gamified cultural tourism platform. The MVP currently centers on:

- one visible destination
- one initial route
- multiple POIs inside that route
- visual objectives attached to those POIs
- progressive hints
- unlockable historical or narrative content

The content model is the first thing that needs durable structure.

This EVO should therefore focus on the read-heavy content backbone of the MVP, not on every future product concern at once.

## Current Repository Reality

At the time of this EVO:

- `apps/api` is the active backend workspace
- the backend runs as a Cloudflare Worker
- D1 has already been provisioned and bound to the Worker as `env.DB`
- a migrations location already exists under `apps/api/migrations/`
- no product schema should exist yet
- no seed data should exist yet

Practical interpretation:

- do not add application query logic
- do not insert Jaen, Catedral, or Banos Arabes data yet
- keep this task focused on a minimal but serious first schema

## Technical Goal

Create an initial D1-compatible SQL schema that is:

- explicit
- relational where it clearly helps
- small enough to evolve safely
- sufficient for the first content endpoints that will come later

This EVO should define only the schema needed to support the first content slice of the MVP.

## Scope

This task should include:

1. review the MVP content model already implied by the repo;
2. define the initial D1 schema in SQL;
3. create the first migration file in the existing migrations folder;
4. use clear table names and column names;
5. add primary keys;
6. add foreign keys where they clarify the content hierarchy;
7. add basic indexes for expected lookup patterns;

9. represent route, POI, objective, and hint ordering explicitly;
10. document the schema briefly;
11. verify that the migration can be applied locally;
12. verify that the migration can be applied to the remote dev D1 database if appropriate;

## Recommended Tables

The initial schema should include tables equivalent to:

```txt
destinations
routes
pois
visual_objectives
hints
unlockable_contents
```

These tables cover the MVP content backbone without prematurely freezing unrelated concerns such as identity, progress, analytics, or demo simulation.

## Recommended Table Responsibilities

### destinations

Stores destination-level content such as Jaen and future destinations.

Expected fields may include:

- `id`
- `slug`
- `name`
- `description`
- `status`
- `cover_image_url`
- `display_order`
- `created_at`
- `updated_at`

### routes

Stores routes belonging to a destination.

Expected fields may include:

- `id`
- `destination_id`
- `slug`
- `title`
- `description`
- `status`
- `difficulty`
- `estimated_duration_minutes`
- `display_order`
- `created_at`
- `updated_at`

### pois

Stores points of interest belonging to a route.

Expected fields may include:

- `id`
- `route_id`
- `slug`
- `name`
- `description`
- `latitude`
- `longitude`
- `access_notes`
- `indoor_mode`
- `display_order`
- `created_at`
- `updated_at`

### visual_objectives

Stores the visual details users must find and later validate.

Expected fields may include:

- `id`
- `poi_id`
- `slug`
- `title`
- `description`
- `target_type`
- `gps_radius_meters`
- `difficulty`
- `display_order`
- `created_at`
- `updated_at`

Reference images can remain as nullable URL/text fields or be deferred entirely until media handling is clearer. Do not overdesign storage around R2 yet.

### hints

Stores progressive hints for each visual objective.

Expected fields may include:

- `id`
- `objective_id`
- `level`
- `text`
- `penalizes_perfect_completion`
- `created_at`
- `updated_at`

### unlockable_contents

Stores content unlocked after a visual objective is completed.

Expected fields may include:

- `id`
- `objective_id`
- `title`
- `short_text`
- `long_text`
- `audio_url`
- `image_url`
- `content_type`
- `display_order`
- `created_at`
- `updated_at`

Audio and image URLs can remain nullable until R2 is introduced.

## Explicitly Deferred Concerns

The following concerns should be handled in later EVOs, not here:

- `users`
- `user_progress`
- `achievements`
- `analytics_events`
- `demo_assets`

Reason:

- they introduce product decisions that are not yet sufficiently stabilized in the repo
- they are not required to define the first content retrieval schema
- adding them now would widen the migration unnecessarily and increase churn risk

## SQLite / D1 Guidance

Cloudflare D1 is based on SQLite, so the schema should stay compatible with SQLite conventions.

Recommended approach:

- use `TEXT` for IDs
- use ISO datetime strings for timestamps
- use `INTEGER` for booleans where needed
- use `REAL` for latitude/longitude
- keep constraints simple
- avoid database-specific features not supported by D1

Do not introduce an ORM in this EVO unless the project has already explicitly adopted one.

## Migration Guidance

Create an initial migration file in the existing migrations folder.

Recommended naming style:

```txt
0001_initial_content_schema.sql
```

The migration should be executable with the repository’s established Wrangler workflow.

Do not add seed inserts to this migration. Product seed data belongs to later EVOs.

## Index Guidance

Add basic indexes for expected access patterns.

Recommended indexes:

- destinations by `status` and `display_order`
- routes by `destination_id`, `status`, and `display_order`
- POIs by `route_id` and `display_order`
- visual objectives by `poi_id` and `display_order`
- hints by `objective_id` and `level`
- unlockable contents by `objective_id` and `display_order`

Do not over-index. The MVP should stay simple.
